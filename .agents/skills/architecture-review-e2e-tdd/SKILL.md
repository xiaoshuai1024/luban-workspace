# Architecture Review + E2E TDD + Security Compliance Skill

本 skill 用于在方案阶段进行架构自检、E2E 测试覆盖分析以及安全合规审查。

## 触发条件

- 新方案编写完成后、执行前
- 用户调用 `/e2e-archi` 命令时
- 方案评审时需要验证测试覆盖度和安全合规

## 输入约定

Agent 需要从上下文获取或向用户索取：

1. **方案文档路径**：指向 `docs/superpowers/plans/*.md`
2. **核心接口设计**：REST API 清单（含路径、方法、说明）
3. **数据模型**：核心表结构（含表名、关键字段）
4. **用户旅程**：关键用户操作路径
5. **安全合规文档**：安全设计、隐私政策、合规声明等

#### 1.5 类型系统一致性（MUST）

当新功能引入或复用了与现有系统**语义不同**的同一字段名（如联合套卡中 `shop_id`/`shopId` 为 VARCHAR，商户系统中为 BIGINT），**必须检查**：

- Service 层是否用了 `rs.getLong("shop_id")` 或 `Long.parseLong(shopId)` 读取该字段？
- Controller 层是否用了 `@PathVariable long shopId` 接收参数？
- 前端 TypeScript 类型中该字段是否应定义为 `string` 而非 `number`？
- 所有涉及该字段的 DB 查询 row mapper 是否统一使用 `getString`？
- 其他使用同一列名 `shop_id` 的关联表（如 `joint_card_shop_listing`、`joint_card_line`）是否同样处理？

**不符合 → 阻断**（运行时必抛 NumberFormatException） |

#### 2.4 测试健康度检查（MUST）

审查每个端（C端/商家端/运营后台）的 E2E 测试时，**必须逐项检查**以下健康度指标，发现问题归入 rubric **阻断** 或 **强烈建议**：

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **test.skip() 占比** | 无环境依赖的测试不应有 `test.skip()`；数据依赖的 skip 应 < 30% 且附日志说明 | **阻断**（> 50% 静默跳过）或 **强烈建议**（30-50%） |
| **环境变量 gate** | 不应有非必要的 `process.env.X !== "1"` gate 阻止整个 describe 运行 | **阻断**（除非该端 E2E 有特殊前提条件且文档注明） |
| **跨端数据自动传递** | 跨端测试间的数据（订单号、退款ID）须通过脚本自动传递，不得仅靠手动设置 env var。**推荐模式**：阶段化 runner（`run-*-cross.mjs`）输出 `HANDSHAKE_*` 标记 → 桥接脚本解析 → 写入临时 env 文件 → 后续阶段读取。跨端测试失败时必须输出统一 context ID 用于串联三端日志定位 | **阻断** |
| **测试重复** | 同一用户旅程不应被 ≥2 个 spec 文件覆盖（同一流程多次调用不同 flow 函数） | **强烈建议** |
| **负向场景覆盖** | P0 流程至少需覆盖：支付失败、操作拒绝、库存不足、鉴权失败中的 2 种 | **强烈建议**（仅有正路径时） |
| **写回验证** | 运营后台 Playwright 不应仅检查页面可达（标题/表格可见），至少需一个数据写回并验证持久化的测试 | **强烈建议** |
| **自动化回滚** | 测试的数据副作用（创建的订单、租户配置等）应在 teardown 中清理，或使用隔离的测试数据 | **建议** |

#### 2.5 低代码引擎渲染专项检查（MUST）

审查低代码引擎/物料/schema 相关 E2E 测试时，**必须检查**以下引擎特有风险点：

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **物料注册覆盖** | 所有已注册物料（luban-ui 组件）是否都有 schema + 渲染 E2E 覆盖？分包/异步加载的物料是否能正确解析注册 | **阻断**（核心物料可达但 schema/渲染从未被 E2E 触碰） |
| **各端渲染一致性** | 同一 schema 在 engine / website(SSR) / electron / flutter 各端的渲染结果是否一致？是否覆盖跨端一致性断言 | **阻断** |
| **props schema 合规** | 物料 props 是否符合 `.agents/rules/luban-material-schema.md`？非法 schema 渲染降级路径是否有 E2E | **阻断** |
| **画布/渲染器版本兼容** | 是否在 CI 中固定了引擎产物版本？引擎升级后是否重新验证了物料渲染 E2E | **强烈建议** |
| **产物构建一致性** | `pnpm run build` 产物在 website SSR 下渲染零新增 console error 是否有断言 | **强烈建议** |
| **schema 热更新** | 物料 schema 变更后页面状态保持/重渲染场景是否有测试 | **建议** |

#### 2.6 测试环境一致性检查（MUST）

**必须检查** E2E 测试环境与生产环境的差异，避免「CI 全绿、生产挂」：

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **Mock 策略对齐** | 第三方 mock（BFF 聚合接口、Java/Go 双后端、OSS）的行为是否与生产真实行为一致？mock 是否过度简化导致遗漏真实逻辑 | **阻断** |
| **双后端配置一致性** | Java 与 Go 双后端的 feature flag、物料 schema 版本是否一致？不一致可能导致测试通过但行为不同 | **阻断** |
| **后端版本匹配** | E2E 测试运行时所连接的后端是否与当前 PR 的代码版本匹配？版本漂移可能导致假红 | **阻断** |
| **数据一致性** | CI 中的测试数据库 schema 是否与生产同步？迁移后测试数据是否同步更新 | **强烈建议** |
| **网络环境差异** | CI 环境的网络延迟、带宽与生产差异是否会影响测试结果（超时、重试行为） | **建议** |

#### 2.7 Flaky Test 治理

**必须检查** E2E 测试的稳定性指标：

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **Flaky 率门禁** | 单次运行 flaky 率（同一用例多次执行结果不一致）不得超过 5% | **阻断** |
| **Flaky 标记机制** | 是否有自动标记 flaky 测试的机制（连续 3 次运行中至少 1 次失败/成功不一致） | **强烈建议** |
| **根因分析时效** | 标记为 flaky 的测试是否在 1 周内完成根因分析并修复或移除 | **强烈建议** |
| **重试策略** | 用例级别的重试是否有明确策略（最多重试次数、重试间隔）？重试后成功是否仍标记为 flaky | **建议** |

#### 2.8 CI 集成检查

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **运行总时长** | E2E 全量运行时间不超过 30 分钟（P0 子集不超过 10 分钟） | **强烈建议** |
| **并行策略** | 测试用例是否按租户/模块并行执行？是否存在资源竞争导致的不稳定 | **强烈建议** |
| **超时处理** | 每个用例是否有合理的超时设置（默认不应超过 60 秒） | **强烈建议** |
| **失败告警** | E2E 失败是否有即时通知机制（飞书/邮件/钉钉）？失败原因是否自动分类 | **建议** |
| **CI 结果报告** | 是否产出可读的测试报告，**必须包含**：失败截图、操作录屏、失败时的页面 DOM 快照、API 请求/响应日志 | **阻断** |
| **失败分类** | CI 失败是否自动分类归类：环境问题 vs 测试问题 vs 代码问题 | **强烈建议** |

#### 2.9 测试数据版本与生命周期管理

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **数据与 schema 对齐** | DB migration（Java Flyway / Go 迁移脚本）后，E2E 测试数据是否同步更新？旧数据 schema 不兼容导致测试失败是否有自动检测 | **阻断** |
| **数据隔离** | 每个 E2E 测试是否使用独立物料工程 / 独立数据范围？并行运行时是否互相干扰 | **阻断** |
| **数据可重复性** | 测试数据是否可重复创建？禁止测试之间通过数据库状态依赖 | **强烈建议** |
| **数据清理** | 测试 teardown 是否清理了创建的测试数据？是否有残留数据随时间累积的问题 | **建议** |

#### 2.10 并发/竞态测试要求

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **并发场景识别** | 方案中是否识别出需要并发测试的场景（秒杀、库存扣减、优惠券领取、排队抢单） | **强烈建议**（关键业务无并发场景识别） |
| **竞态条件覆盖** | 已识别的并发场景是否有对应的 E2E 测试（同时发起请求、验证数据一致性） | **阻断**（秒杀/库存类无竞态测试） |
| **幂等验证** | 重复提交（支付、下单、退款）是否有测试覆盖？重复请求应返回相同结果而非异常 | **阻断** |
| **超时与锁超时** | 分布式锁超时、数据库行锁等待超时的行为是否有异常路径覆盖 | **建议** |

#### 2.11 Feature Flag 组合测试

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **Flag 组合矩阵** | 多租户 feature flag 的组合是否形成测试矩阵？至少覆盖：全开、全关、关键 flag 单独开启 3 种组合 | **强烈建议** |
| **灰度场景** | 灰度发布的 flag 切换是否有 E2E 验证（开启前→开启后→关闭后，页面和 API 表现正确） | **强烈建议** |
| **AB 测试一致性** | AB 测试的分组逻辑是否在 E2E 中可重现？不同分组应返回预期不同的 UI/数据 | **建议** |

#### 3.4 测试代码自身质量审查（MUST）

审查 MR 中的新 E2E 测试时，**必须逐项检查**：

| 检查项 | 验收标准 | 不符合时 |
|--------|----------|----------|
| **选择器规范** | 是否统一使用 `data-testid` 属性？禁止使用 CSS class 或 text 内容做选择器 | **阻断** |
| **断言完整性** | 每个测试是否包含至少一个真实断言（禁止 `expect(true).toBe(true)` 占位断言） | **阻断** |
| **数据清理** | 测试创建的副作用数据是否有 teardown 清理或使用隔离范围 | **阻断** |
| **测试独立性** | 测试是否可独立运行？是否有全局状态依赖（共享变量、未清理的 mock） | **强烈建议** |
| **复杂度控制** | 单用例行数不超过 100 行？一个 test 块做且只做一件事（名称中不应含 "and"） | **强烈建议** |

#### 3.5 方案阶段外 E2E 质量门禁

**合入门禁阶段**：
- E2E P0 测试必须全绿才能合并
- E2E 运行总时间不能超过 §2.8 约定的阈值
- 新增 E2E 测试必须通过 §3.4 的代码质量审查

**上线前回归阶段**：
- 全量 E2E 回归运行
- 对比上次运行的 diff 分析（新增失败、新增跳过、运行时间变化）
- flaky 率检查（参考 §2.7）

**线上监控阶段**：
- 线上问题与 E2E 测试的关联分析（已覆盖的旅程不应再出现线上故障）
- 线上故障触发后，补写对应场景的 E2E 测试
