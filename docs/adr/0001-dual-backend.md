## 背景 (Context)

Luban 后端需要同时支撑两类负载：一类是复杂业务逻辑（物料、表单、线索、状态机、事务边界密集），强调生态成熟度与可维护性；另一类是高 QPS 的网关式访问（公开站点、健康检查、鉴权），强调延迟与资源占用。单一语言很难在两端都做到最优：Java 在企业级 ORM/事务上生态深厚，但常驻内存与冷启动偏高；Go 在并发与单机资源占用上优势明显，但企业级中间件与事务抽象相对薄弱。

另一方面，供应链风险已成为现实威胁。Log4Shell 一类事件表明，单一技术栈的单一依赖链一旦出现严重漏洞，整个后端会同时停摆，没有可快速切流的备用实现。

基于以上两点，项目早期就面临一个选择：后端到底用一种语言实现一次，还是用两种语言各实现一次、共享同一套业务契约。

## 决策 (Decision)

同一业务用 Java（Spring Boot + MyBatis + Flyway）和 Go（Gin）双实现，所有接口契约两端同步维护。

具体落地：

- **契约维度**：路径、请求参数（字段名/类型/必填性）、请求体、成功响应体、错误响应体（错误码一致，message 措辞可异）、HTTP 状态码、业务状态机、副作用（DB/缓存/事件）、幂等性——九个维度两端必须对齐。
- **同步变更硬约束**：同一 PR / 同一任务内完成两端改动，禁止只改一端；差异化实现须在 PR 中显式标注理由。
- **角色分工**：Java 作主实现承载复杂业务逻辑；Go 作高 QPS / 网关路径与灰度承载。BFF 作为统一入口，抹平差异而非掩盖差异（字段映射、错误码归一可做；补齐缺失字段、改业务语义不可做）。
- **现状**：Java 是主实现（端口 8080，context-path `/backend`）；Go 是部分实现（端口 8081，容器内 8080），已覆盖 `/ping`、`/public/sites`、`/auth`、`/sites`、`/users`、`/settings`、`/datasources`，尚缺 `/leads`、`/forms`。Go 的缺口作为已知项单独跟进。
- **自动化守护**：Java 端用 ArchUnit、Go 端用手写 ast 测试做对称契约守护；并维护一份双后端 contract test，断言响应等价。

## 考虑过的备选方案 (Alternatives Considered)

### 备选 A：单 Java 后端
- 优点：实现成本最低，只维护一份代码；Spring 生态成熟，招人与上手快；契约对齐问题天然消失。
- 缺点 / 代价：高 QPS 网关路径上 Java 的内存占用与延迟不占优；供应链单点风险无法消除（一次 Log4Shell 级事件即全停）；未来若想按场景路由到更轻量的运行时没有现成实现，迁移成本会随业务增长而升高。

### 备选 B：单 Go 后端
- 优点：性能与资源占用优秀，部署体积小、冷启动快；高并发场景天然契合。
- 缺点 / 代价：复杂业务逻辑（多表事务、ORM、领域事件）需要大量自建抽象，企业级生态与中间件不及 Java；同样存在供应链单点风险；团队若以 Java 背景为主，Go 在复杂业务上的交付速度与正确性风险更高。

### 备选 C：双语言但只按场景各做一部分（不要求契约对齐）
- 优点：两端各做擅长的部分，开发量小于「双实现」。
- 缺点 / 代价：失去互为热备 / 故障转移的能力（一端挂了无法切到另一端）；BFF 难以做灰度与降级路由；接口归属一旦划分，业务演进时容易出现「这个接口该归谁」的反复拉扯，治理成本反而更高。

## 后果 (Consequences)

- **正面**：供应链单点风险被双实现有效稀释；可按场景与灰度策略在两端间路由流量，故障转移成为可能；技术栈多样性为团队保留演进空间。
- **负面 / 代价**：所有接口契约两端同步维护，同一改动的工作量约等于单后端的 1.8–2 倍；Go 实现滞后于 Java（如 `/leads`、`/forms` 暂缺）构成已知契约缺口；需要 ArchUnit + Go ast + contract test 三层自动化守护，CI 复杂度上升。
- **需要后续跟进**：补齐 Go 端 `/leads`、`/forms` 等缺口接口；将 contract test 纳入 CI 门禁；持续校验大整数（字符串传输）、空值语义（null vs 零值）、时间格式（ISO 8601）等高频差异点。

## 备注

推翻本决策的前提：若未来评估双维护成本长期高于其风险对冲价值，应新写一篇 ADR（标 Superseded by）将后端收敛为单一实现，并保留本篇作为历史依据。详细执行规范不在本 ADR 重复，见 `docs/DUAL_BACKEND_PARITY.md`。
