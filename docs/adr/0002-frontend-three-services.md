## 背景 (Context)

luban 前端面向两类截然不同的使用者：内部运营（站点/页面/线索/用户后台 + 全屏设计器 `/designer`）与外部访客（浏览已发布页、提交留资）。这两类使用对运行时形态的要求几乎是相反的：运营侧要的是一个高度交互、状态复杂、长会话的富应用，设计器尤其依赖客户端路由与大量交互态；访客侧要的则是首屏快、可被搜索引擎抓取、可被社交平台预览的静态化内容。把两者塞进同一种渲染模型，必然有一方被牺牲。

同时，后端是 Java 服务（`8080`），存在鉴权、聚合、公开提交等横切关注点。如果让前端各应用各自直连后端，鉴权逻辑（JWT、token 刷新、登录态）就会在多个仓里重复实现并各自漂移；公开留资接口还面临 SSRF 与未鉴权提交的防护需求，需要一个统一收口点。

运维上，三类负载的扩缩容节奏也不同：访客流量大、读多写少、可缓存；运营并发低但会话重；BFF 是无状态转发层。混在一起部署会互相拖累。因此前端在形态、关注点、运维三个维度都需要拆分。

## 决策 (Decision)

我们选择将前端拆为三个独立服务，各自独立端口与部署：运营后台 SPA、聚合 BFF、访客 SSR 站点。

具体形态：**engine** 是 Vue 3 + Vite SPA，端口 `5173`，承载运营后台与全屏设计器 `/designer`，`/api` 经 Vite proxy 转发到 BFF，鉴权用 `localStorage` 的 `luban_token` + axios Bearer 拦截器。**bff** 是 Next.js 16，端口 `3100`（启动须显式 `-p 3100`，避免与 website 的 `3000` 冲突），通过 `BACKEND_BASE_URL` 指向 Java 后端，后端超时 `15000ms`，集中负责 JWT 鉴权、后端聚合、公开 submit 端点与 SSRF 防护（`ssrfGuard`）。**website** 是 Nuxt 3 SSR，端口 `3000`，读取 `bffBaseUrl` 把留资等写操作 POST 到 BFF，渲染已发布页面。

这样 engine 与 website 都不再各自直连 Java 后端，鉴权与聚合只在 BFF 一处实现，由它统一收口。

## 考虑过的备选方案 (Alternatives Considered)

### 备选 A：单一 Next.js 全栈应用（BFF + SSR + 后台合一）
- 优点：单仓单部署，鉴权/聚合/渲染同进程，心智与运维最简单；SSR 与 API 路由天然同框架。
- 缺点 / 代价：后台是强交互富应用，强行用 Next 的 SSR/路由模型承载设计器这类纯客户端交互场景会很别扭，水合成本与状态管理复杂度上升；访客的可缓存 SSR 与运营的长会话共享同一部署与扩缩容，互相拖累；任何一处变更（如设计器改动）都可能触发整站重新部署，回滚粒度过粗。

### 备选 B：单体 Vue SPA（后台 + 访客同一 SPA）
- 优点：技术栈统一为 Vue，团队上下文单一，物料与设计器可直接复用同一套组件体系。
- 缺点 / 代价：访客站点彻底失去 SSR，首屏与 SEO 严重劣化，对一个面向公开访客、依赖自然流量的站点是硬伤；后台鉴权 token 与访客公开页混在同一 SPA，安全边界模糊；访客每次都要下载整套运营后台代码，体积浪费。

## 后果 (Consequences)

- **正面**：每种服务都用最贴合其场景的技术形态（SPA 富交互 / SSR 首屏 / BFF 聚合）；鉴权与后端聚合在 BFF 单点实现，杜绝重复与漂移；三服务可独立扩缩容、独立部署与回滚；engine 与 website 解耦后互不影响发布节奏。
- **负面 / 代价**：三套技术栈（Vue3 / Next16 / Nuxt3）带来更高的认知与维护成本；跨服务联调需同时起三个进程并注意端口约定（BFF 必须 `-p 3100`）；鉴权链路从「直连后端」变成「前端 → BFF → 后端」两跳，多一层需排错。
- **需要后续跟进**：统一三服务间的契约（错误码、分页结构、token 传递）；补一份本地三服务并行启动脚本，固化端口约定；明确 BFF 聚合层的缓存与超时策略（当前后端超时 `15000ms`）。

## 备注

推翻此决策需同时满足：访客站点不再需要 SEO/首屏，或鉴权/聚合被下沉到后端网关层统一接管。与 [ADR 0004](./0004-lowcode-schema-driven.md)（schema-driven）相关：engine 设计器产出的 schema 由 website 渲染，三服务分离是 schema 多端渲染的部署前提。
