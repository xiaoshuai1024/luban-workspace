## 背景 (Context)

luban 的客户端不只一个：除 web 站点（`packages/web/luban-website`）外，还有 `packages/client/luban-electron`、`luban-flutter`、`luban-cross-plateform`。同一批用户会在不同端完成同一件事（登录、创建、编辑、导出），他们对产品的认知是「同一个 luban」，而不是「web 版功能多、桌面版功能少」。

理论上这由低代码引擎兜底：引擎驱动多端渲染，同一份 schema 在各端应产出一致的业务表现。但实际各端运行时差异显著——Web 是最新 Chromium、Electron 是嵌入版本（可能落后、带 Node 集成）、Flutter 走各平台 WebView（iOS WKWebView 与 Android Chromium 版本碎片化）。这些差异天然会把「理论一致」撕出分叉。如果不在架构层立规矩，各端就会各自为政：web 有「导出」按钮、electron 没有；同一状态枚举在 web 叫「已发布」、在客户端叫「上线」；登录在 web 是 3 步、在客户端被「优化」成 2 步。这些不是样式问题，是产品割裂。

问题在于：哪些差异是「平台习惯」可以容许，哪些是「功能缺失」必须对齐，需要一条清晰的判定线，否则各端实现时会以「体验优化」之名行「省事砍功能」之实。

## 决策 (Decision)

我们选择由低代码引擎驱动多端渲染同一 schema 来实现业务一致性，并明确四项不可妥协的要求；功能本身必须各端齐备，只有「实现机制」允许按平台习惯调整。

四项要求：(1) **业务逻辑一致**——同一业务能力的可用功能集、状态机、权限边界、错误处理各端一致，不得因技术栈差异省略功能（如 web 有「导出」而 electron 没有，禁止）。(2) **数据契约一致**——各端消费同一套 BFF API，不各端自造接口，字段映射、错误处理、分页结构一致。(3) **UI 表现一致**——信息架构、操作入口、状态展示一致，容许因平台习惯的样式微调（如 macOS 交通灯位置），但不容许功能缺失。(4) **用户旅程一致**——同一任务各端步骤数与关键节点一致（登录 web 是 3 步，客户端也必须是 3 步）。

判定线由此清晰：功能/契约/旅程层面各端必须对齐；只有窗口控制、文件操作、通知、快捷键、离线等实现机制允许按平台习惯实现（「导出」功能要有，至于走下载还是 fs 由平台定）。

## 考虑过的备选方案 (Alternatives Considered)

### 备选 A：各端独立实现（牺牲一致性换各端最优体验）
- 优点：每端可发挥原生优势，交互、动效、性能都可做到该平台最优；团队可按端并行迭代、互不阻塞。
- 缺点 / 代价：直接违背「同一个 luban」的用户认知，功能矩阵散乱（某能力只在某端有），用户跨端迁移成本高；同一业务逻辑在 N 个端各写一份，维护与回归成本随端数线性放大，状态枚举、错误处理、API 字段必然漂移；测试要为每端单独造用例，多端交叉验证无从谈起。对一个以「引擎统一渲染」为核心卖点的平台，这是自相矛盾。

### 备选 B：仅 web 端 + 客户端套壳 WebView
- 优点：一致性「免费」达成——客户端只是个浏览器壳，渲染逻辑只有 web 一份，零分叉；开发与维护成本最低。
- 缺点 / 代价：彻底放弃客户端原生体验（窗口控制、全局快捷键、完整文件系统、离线、系统通知全都要妥协或丢失）；Flutter 端的目标（跨移动平台原生）根本无法用套壳达成；嵌入 WebView 本身又引入版本碎片化与性能问题，等于把一致性的代价转嫁到体验上，且移动端体验劣化不可接受。

## 后果 (Consequences)

- **正面**：用户跨端体验连贯，产品认知统一；「功能齐备、机制按平台」的判定线让各端实现有据可依，不再以体验之名砍功能；同一 BFF 契约成为各端共同事实源，减少接口漂移；引擎驱动让 schema 层面的一致性可被自动验证。
- **负面 / 代价**：新增能力必须各端同步落地，交付节奏受最慢端拖累；多端运行时差异（CSS 子集、WebView 版本）要求关键样式做多端交叉验证；多端 E2E 测试矩阵成本上升。
- **需要后续跟进**：每个核心业务能力至少在 2 端做交叉验证，关键旅程（登录/创建/编辑/删除）做多端 E2E；engine 渲染 E2E 至少覆盖 web + electron；状态枚举的各端中文映射纳入检查清单。

## 备注

推翻需证明：存在比「引擎驱动多端」更优的一致性保证机制，或产品定位明确放弃某端的原生体验。允许的平台差异（窗口控制/文件/通知/快捷键/离线）见 [luban-multi-client-consistency.md](../../.agents/rules/luban-multi-client-consistency.md) 各端差异合理化表。依赖 [ADR 0004](./0004-lowcode-schema-driven.md)：一致性的事实基础是同一 schema + 同一物料契约。
