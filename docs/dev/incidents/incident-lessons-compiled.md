# 事故复盘经验集（通用教训）

> 从历史事故中提取的通用教训，去业务细节。每条都是可复用的工程经验。
> 适用：luban 全栈，避免同类问题再发生。

---

## 教训一：UI 假绿 — 列表 API 500 未被 E2E 拦截

### 现象
某运营列表页后端返回 HTTP 500，既有 Playwright E2E 用例**全部通过**，形成 UI 假绿。用户实际看到空表 + 错误提示。

### 根因（实现层）
1. **JdbcTemplate 变参误用**：`queryArgs` 为 `List<Object>`，传入 `jdbcTemplate.query(..., queryArgs)` 时 Java 将其当作**单个**绑定参数，未展开 `LIMIT`/`OFFSET` 的 `?`。
2. **路由冲突**：未注册静态路径 `/rules`，被 `/{id}` 贪婪匹配，`id=rules`，仅允许对应 method → 405/500。

### 根因（测试与流程）
1. **E2E 只验壳、不验 API**：用例断言表格/标题**可见**；接口 500 时前端仍渲染空表，错误提示不导致用例失败。
2. **后端 IT 未覆盖分页绑定**：集成测试调用列表接口但**未传**分页参数，未执行带 `LIMIT ? OFFSET ?` 的代码路径。
3. **新接口无契约测试**：前端页面已上线，后端未同步对应端点，无 API 级回归。

### 通用教训
1. **列表页 E2E 必须断言关键 API 200 + 业务 code**，禁止仅断言 UI 元素可见（见 `docs/dev/e2e-test-style-guide.md` §10）。
2. **JdbcTemplate 动态参数必须用 `Object[]` 重载**，禁止把 `List` 传给变参 `query(..., Object... args)`。
3. **静态路径必须放在 `/{id}` 之前**或独立 `@RequestMapping`，避免贪婪匹配。
4. **后端 IT 必须覆盖带分页参数的 SQL 路径**。
5. **前后端契约**：前端调用的端点必须有后端对应实现，CI 加跨层契约检查。

## 教训二：批量推送脚本误推保护分支

### 现象
执行批量提交脚本（`/pr-all` 等价物）时，某子仓处于 `dev` 分支被自动 commit + push 到远程 dev，commit message 为当前主仓 MR 标题而非实际变更内容。

### 根因
批量脚本的「all」目标在遍历子仓时**未做分支检查**：
- 单仓函数内有 `branch_ok()` 检查，但「all」目标内联了 commit+push+MR 逻辑，遗漏了分支校验。
- 子仓在 `dev`/`master`/`main` 等保护分支时仍被推。

### 后果
- 保护分支被推了 commit message 不匹配的提交
- MR 创建阶段因源分支=目标分支返回 400，脚本中断
- 其他子仓未完成 MR 创建

### 通用教训
1. **批量脚本的「all」目标必须与单仓函数保持一致的校验**，禁止内联重复逻辑绕过检查。
2. **分支白名单**：仅 `feature/*` / `bugfix/*` / `hotfix/*` 才允许 commit+push+MR，保护分支（`dev`/`master`/`main`）一律跳过。
3. **保护分支额外防护**：即使脚本跳过，git 端 pre-receive hook 也可兜底，禁止向保护分支直接 push。
4. **重构建议**：让「all」目标也调用单仓函数，而非内联重复逻辑。

## 教训三：修复被后续提交冲掉（编译通过 ≠ 测试通过）

### 现象
某模块大量功能损坏：后端并发保护代码（CAS 守卫、FOR UPDATE、参数化 SQL）被后续优化提交从旧基线覆写删除；前端调用后端从未实现的端点；合约检查脚本被误删。review 的 317 项修复加了并发保护和前端调用，但 3 天后全部被冲掉。

### 根因
1. **review 只做编译验证不做测试**：修复 agent 完成后只跑 `mvn compile` / `pnpm build`，不跑 `mvn verify` / `pnpm test`。编译通过就自动提交，删掉的代码、调不存在的端点全漏了。
2. **无跨仓合约检查**：review 逐仓审查，前端调后端不存在的端点在前端仓内看不出来。
3. **自动提交无人工审核**：317 项修复一口气提交，个别引入新问题无法逐条拦截。
4. **合约检查脚本被删后无兜底**：删除后没有任何机制检测跨仓契约不一致。

### 通用教训
1. **review 提交前必须通过测试门禁**（`testGatePassed == true`）。
2. **修复 agent 增加定向测试层**：改了 `XxxService` → 跑 `XxxServiceTest`。
3. **合约检查纳入 git pre-commit hook 和 review 流程**。
4. **E2E 为硬门禁，禁止跳过、禁止占位测试**。
5. **关键原则**：`mvn compile` 通过 ≠ 代码正确。只有 `mvn verify`（含单测+集成测+覆盖率门禁）才能确认代码质量。
6. **分支冲突保护**：大改动用 worktree isolation，合并时检查是否冲掉关键修复。

## 教训四：多数据源 SQL 隔离审查

### 场景
review 发现多处 SQL 查询缺少上下文隔离条件（如多租户 `tenant_id`、软删除 `delete_time`），导致跨上下文数据泄露。

### 通用教训
1. **所有包含上下文实体表的 JOIN 都必须有隔离条件**（`tenant_id = ?` / `delete_time = 0`）。
2. **检查参数数组长度与 SQL 中 `?` 数量一致**。
3. **验证方法参数确实被传递到 SQL 参数字面量中**（不是只声明不使用）。
4. review 的 agent prompt 必须包含数据隔离检查清单。

### luban 特化
- 双后端的隔离逻辑必须一致：Java 和 Go 同一查询的 WHERE 条件必须一致。
- 引擎/BFF 层的权限校验不能替代后端的 SQL 隔离（纵深防御）。

## 通用事故预防清单

每次合入前自检：

- [ ] E2E 是否真实断言（非假绿）？列表页是否断言关键 API 200？
- [ ] 批量脚本是否做了分支白名单检查？
- [ ] review 是否通过了测试门禁（非仅编译）？
- [ ] 跨仓/跨层契约是否检查？
- [ ] SQL 查询是否都有上下文隔离条件？
- [ ] 保护分支是否禁止直接 push？
- [ ] 关键修复是否被后续合并冲掉？

## luban 特化预防

- 双后端（Java/Go）契约一致性是硬门禁（`docs/DUAL_BACKEND_PARITY.md`）。
- 多端（electron/flutter/web）业务一致性。
- 引擎/物料 schema 合规、渲染零 console error。
- 11 子仓的跨仓 API 路径 diff 检查。
