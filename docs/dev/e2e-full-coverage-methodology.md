# 新模块 E2E 全流程覆盖方法论

> 当一个新模块包含多个独立流程时，如何确保每个流程都被 E2E 测试覆盖、不遗漏。通用方法论，适用 luban 任意业务模块。

## 方法论：流程拆解 → 覆盖矩阵 → 补齐 → 验证

### 第一步：流程拆解（Flow Inventory）

深入代码，识别出模块中所有**独立的用户可操作流程**。每个流程应满足：

- 有明确的入口（API 端点 或 页面路由）
- 有可观测的输出（DB 变更 / 第三方调用 / 页面状态变化）
- 可以独立运行，不依赖其他流程的结果

**输出格式**（流程清单表）：

| # | 流程 | 类型 | 入口 | 可观测输出 |
|---|------|------|------|-----------|
| ① | 基础设施/接入层 | 基础设施 | POST /api/... | DB 状态 |
| ② | 实体创建 | 独立 | POST /api/... | 实体记录 |
| ③ | 实体查询 | 独立 | GET /api/... | 列表/详情 |
| ... | ... | ... | ... | ... |

### 第二步：覆盖矩阵（Coverage Matrix）

对每个流程，检查测试层次覆盖情况：

```
┌──────────────┬──────────┬──────────┬──────────┬──────────┐
│ 流程          │ 后端 IT  │ 引擎/BFF │ 多端 E2E │ 跨层契约  │
├──────────────┼──────────┼──────────┼──────────┼──────────┤
│ ① 接入层      │ ✅/❌    │ N/A      │ N/A      │ ✅/❌    │
│ ② 实体创建    │ ✅/❌    │ ✅/❌    │ ✅/❌    │ ✅/❌    │
│ ...           │ ...      │ ...      │ ...      │ ...      │
└──────────────┴──────────┴──────────┴──────────┴──────────┘
```

### 第三步：补齐策略（Gap Filling）

按优先级补齐：

1. **后端集成测试（IT）** — 验证 Controller → Service → DB 全链路，外部依赖 mock
   - 每个流程至少 1 个成功场景 + 1 个异常场景
   - Mock 配置需要完整覆盖所有外部 API 调用
   - Java 和 Go 双后端都要写

2. **引擎/BFF E2E** — 验证渲染/BFF 聚合正确性
   - 物料 props schema 合规
   - BFF 聚合字段与后端返回一致
   - 引擎产物渲染一致

3. **多端（electron/flutter/web）E2E** — 验证各端业务一致
   - 页面加载验证（URL、标题、关键元素）
   - 操作交互验证
   - 有条件执行（仅当数据存在时测试操作）

### 第四步：验证门禁（Verification Gate）

- 后端：`mvn -q verify`（Java）/ `go test ./...`（Go）编译 + 测试通过
- 引擎/UI：`pnpm test && pnpm run build` 通过
- E2E：`pnpm run test:e2e` 全部通过（无 skip、无假绿）

## 关键经验

### Mock 配置要全面

每个外部依赖（第三方 API、消息队列、文件存储等）都要 mock 完整，否则 IT 测试会因真实调用失败而误判。

### E2E 测试要容错

E2E 运行在真实后端上，数据状态不确定。使用条件执行模式：

```typescript
const btn = page.locator("button:has-text('xxx')").first();
const exists = (await btn.count()) > 0;
if (exists) {
  await btn.click();
  // ...验证交互
}
```

但**仅用于非核心路径**。核心写操作（创建/状态流转）必须有确定性数据，不能用条件执行。

### 跑通是底线

「补齐」不仅写文件，还要跑通。存量测试失败也要一并修复，否则每次 CI 都红。

### 跨层契约必须验证

前后端由不同 agent/人开发时，类型契约容易不一致（字段名、类型、枚举值）。每个模块的 E2E 必须包含跨层契约断言：
- 前端调用的字段名 = 后端返回的字段名
- 类型一致（如 id 是 string 还是 number）
- 枚举值集合一致

## luban 特化

- **双后端契约**：Java 和 Go 同一接口的响应体必须一致（见 `docs/DUAL_BACKEND_PARITY.md`）。
- **引擎渲染**：物料 schema 变更后，引擎渲染 E2E 必须覆盖（见 `docs/LOWCODE_ENGINE_SPEC.md`）。
- **多端一致**：同一业务在 electron/flutter/web 三端行为一致（见 `.agents/rules/luban-multi-client-consistency.md`）。
