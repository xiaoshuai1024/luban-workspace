# 压测方法论（通用）

> 上线前压测的通用方法论。原应用于某次大规模压测方案，提炼为可复用框架，去业务化（不绑定具体用户数与业务场景）。
> 适用：luban 各后端（Java/Go）、BFF、引擎 SSR、website 的容量验证。

---

## 1. 压测目标

### 1.1 容量推算（按目标用户规模）

| 指标 | 计算公式 | 示例 |
|------|---------|------|
| 注册用户 | 目标 | N |
| DAU（日活） | 注册 × 30% | 0.3N |
| 峰值并发（PCU） | DAU × 15% | 0.045N |
| 峰值 QPS | PCU × 5 请求/秒 | 0.225N |
| 峰值写 QPS | PCU × 1 写/秒 | 0.045N |

> 常规压测目标定在 **2× 预期峰值并发**。高并发抢购/秒杀场景单独设计。

### 1.2 通过标准

| 指标 | 阈值建议 | 严重程度 |
|------|---------|---------|
| P95 响应时间（纯读接口） | ≤ 800ms | 警告 |
| P95 响应时间（写接口） | ≤ 2,000ms | 警告 |
| P99 响应时间（所有接口） | ≤ 3,000ms | 严重 |
| 错误率 | ≤ 0.1%（不含 429） | 严重 |
| 超时率 | 0% | 阻塞 |
| CPU 使用率 | ≤ 80% | 警告 |
| 内存使用率 | ≤ 80% | 严重 |
| 数据库连接池使用率 | ≤ 70% | 警告 |
| 慢查询（>阈值） | 为 0 | 严重 |
| 高并发抢购：库存一致性 | 零超卖 | 严重 |

（具体阈值按业务 SLA 调整）

## 2. 压测环境

| 选项 | 说明 | 推荐 |
|------|------|------|
| **预发布（Staging）** | 模拟生产配置，独立数据库 | ✅ 首选 |
| 生产（避开业务高峰） | 真实环境，有风险 | 仅上线前烟雾测试 |
| 本地开发 | 配置差异大，结果不可靠 | ❌ 不适用 |

搭建与生产一致的 **Staging 环境**：
1. 服务器规格与生产一致
2. 数据库从生产导出脱敏数据
3. Redis、ES 独立实例
4. 压测流量从独立压测机发出（避免压测工具本身成为瓶颈）
5. 高并发场景需额外准备 Redis 独立实例

## 3. 压测工具选择

| 工具 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **JMeter** | 全链路复杂场景 | 支持断言/关联/分布式/GUI/图表 | 资源消耗大 |
| **k6** | 脚本化压测 | JS 脚本/低资源/CI 集成 | 需要写 JS |
| **wrk** | 单接口快速压测 | 极轻量/高并发 | 不支持复杂场景 |
| **vegeta** | Go 生态简单场景 | 高性能 | 功能有限 |

**推荐组合：JMeter（全链路）+ wrk（单接口快速验证）**

## 4. 压测场景设计

### 4.1 用户行为模型

定义加权混合场景，权重基于真实用户行为分布：

| 场景 | 类型 | 权重 | 说明 |
|------|------|------|------|
| 列表浏览 | 读 | 高 | 主要读流量 |
| 详情查询 | 读 | 中 | 含搜索/筛选 |
| 创建/下单 | 写 | 低 | 写流量 |
| 状态流转 | 写 | 低 | 关键写操作 |
| 搜索 | 读 | 中 | 依赖搜索引擎 |
| 高并发抢购 | 写（burst） | 极低频但高并发 | 单独设计 |

### 4.2 场景类别（通用）

| 场景 | 类型 | 说明 |
|------|------|------|
| A | 纯读浏览 | 多模块混读，验证读链路 |
| B | 混合写流程 | 创建 → 确认 → 查询，验证写链路 |
| C | 高频交互 | 点赞/评论/收藏等轻量写 |
| D | 资金/敏感操作 | 退款/提现等，权重低但要求高一致性 |
| E | 特殊业务专测 | 按业务特性（如核销、激活） |
| F | 高并发抢购（专项） | 验证库存扣减、热点行锁、缓存击穿 |
| 稳定性测试 | 耐力 | 70% 峰值并发，持续 30 分钟 |
| 极限测试 | 容量 | 梯度递增找拐点 |

### 4.3 稳定性测试（耐力测试）

| 参数 | 值 |
|------|-----|
| 并发用户 | 70% 峰值 |
| 混合场景 | 各类场景按权重 |
| 持续时间 | 30 分钟 |
| 观察项 | 内存泄漏、连接池耗尽、慢查询累积、GC 频率、搜索响应退化 |

### 4.4 极限测试（容量测试）

梯度递增，找到系统拐点：

```
100 → 300 → 500 → 1000 → 1500 → 2000 → 3000 并发
每个梯度持续 1-3 分钟，观察错误率 >1% 或 P99 >5s 的拐点
```

### 4.5 高并发抢购专项（如有）

三阶段：
1. **预热（Phase 1）**：大量用户浏览目标商品（500 并发，30s）
2. **爆发（Phase 2）**：并发抢购下单（1000+ 并发，5s burst）
3. **回稳（Phase 3）**：查单及回退流量（200 并发，30s）

**关键关注点**：
- 库存超卖：压测后验证 `库存扣减数 == 订单成功数`
- 热点行锁：同一商品 ID 的行锁竞争
- 缓存击穿：热点缓存失效瞬间对 DB 的压力
- 重复下单：幂等性机制是否有效

## 5. 测试数据准备

### 5.1 数据量要求

按目标用户规模准备种子数据：

| 数据表 | 建议量 |
|--------|--------|
| 用户 | 目标用户数 |
| 核心实体 | 每用户 N 条 |
| 历史记录 | 覆盖多种状态 |
| 热点数据 | 10+ 个用于高并发场景 |

### 5.2 种子数据生成策略

用存储过程 + `INSERT INTO ... SELECT` 批量生成，避免逐条插入。

### 5.3 认证 Token 池

需要鉴权的接口，提前生成 token 池：
- 提前生成 2× 峰值并发数的有效 token
- 存入 Redis 或文件供压测工具使用

### 5.4 高并发场景特殊数据

| 项目 | 配置 |
|------|------|
| 抢购目标数 | 5-10 个集中热点 |
| 每个目标库存 | 500-1000（验证真实扣减不超卖） |
| 预热缓存 | 压测前预热 Redis + 搜索引擎 |
| 幂等键 | 每请求携带唯一 key |

### 6.2 数据库实时监控

```sql

```


## 7. 压测执行步骤

### 第一阶段：基线 — 单接口 + 纯读场景
- 搭建 Staging，导入种子数据
- 预热缓存与搜索引擎
- 配置监控面板
- wrk 单接口基线，记录 P50/P95/P99
- 纯读场景 500/1000/2000 并发

### 第二阶段：全链路压测 — 写场景 + 稳定性
- 混合写场景各并发梯度
- 稳定性测试 30 分钟
- 梯度极限测试找拐点

### 第三阶段：高并发专项 + 优化
- 高并发抢购场景
- 验证库存一致性、幂等性
- 未通过则定位瓶颈、针对性优化、重测

### 第四阶段：优化迭代
- 定位瓶颈（看 Grafana：JVM/DB/Redis/搜索引擎哪个先到极限）
- 针对性优化
- 重新压测，记录优化效果

## 8. 瓶颈分析与优化预案

| 瓶颈点 | 症状 | 提前检查 | 优化方案 |
|--------|------|---------|---------|
| **数据库** | 连接池打满、慢查询、死锁 | `SHOW PROCESSLIST` | 加索引、SQL 优化、读写分离 |
| **热点行锁** | 高并发 INSERT/UPDATE 等待 | `SHOW ENGINE INNODB STATUS` | Redis 预扣 + 异步回写、库存独立表 |
| **JVM 堆** | GC 频繁、Full GC、OOM | `jstat -gcutil` | 调大 `-Xmx`、排查内存泄漏 |
| **线程池** | 请求排队 | threads metrics | 调大 max threads、异步化 |
| **Redis** | 命令超时、内存打满 | `INFO stats` | 优化缓存策略、集群分片 |
| **搜索引擎** | 查询慢、节点负载高 | `_cat/nodes` | 优化 mapping、结果缓存 |
| **网络** | 带宽打满 | `iftop` | 升级带宽、CDN、压缩响应 |

### 高并发优化建议

1. **Redis 预扣库存**：下单前在 Redis 用 `DECR` 扣减，仅正值后才允许下单，避免热点行锁
2. **Lua 脚本原子扣减**：防止超卖
3. **下单异步化**：请求入消息队列，异步消费，前端轮询
4. **静态化页面**：CDN 静态化减少应用服务器压力
5. **限流降级**：接口限流，超出返回「拥挤」文案

## 9. 风险与注意事项

1. **数据污染**：压测数据不应与生产混合；压测后用 `TRUNCATE` 清理测试表
2. **第三方支付**：压测时无法调用真实支付，必须开启 mock
3. **第三方登录**：压测环境应提供 mock token 生成，绕过真实登录
4. **搜索引擎**：确保已索引压测数据，避免首次查询慢
5. **幂等性**：写操作须具备幂等性，避免重复执行
6. **可预期业务拒绝**：高并发中部分请求会产生预期的 403/422，需在错误率统计中排除

---

## 10. luban 特化

- **双后端压测**：Java 和 Go 后端都要压，验证两者性能在同一量级（见 `docs/DUAL_BACKEND_PARITY.md`）。
- **BFF 聚合层**：BFF 聚合多个后端接口时，聚合耗时 = max(各后端耗时) + 聚合开销，需单独压测。
- **引擎 SSR**：luban-website 用 SSR，SSR 渲染耗时受物料复杂度影响，需覆盖复杂页面。
- **多端一致**：压测目标按各端的预期并发分别设定。
