# 认证与安全策略

> 通用鉴权/安全实践，适用于 luban 多端（engine/bff/website/electron/flutter/web）+ 双后端（Java/Go）。业务无关。

## 一、凭证交换原则：禁止 mock/fallback，禁止静默降级

任何涉及第三方凭证交换的登录/接入流程（OAuth、SSO、外部 IdP 等）：

- **必须**使用真实凭证调用第三方接口
- **禁止**使用 mock/fallback 标识，包括本地开发环境、CI 环境
- 凭证缺失时，登录接口直接返回 503 并提示「凭证配置不完整，请联系运维配置」
- 第三方调用失败时（凭证不匹配 / token 无效），直接抛出原异常，**不静默降级**

### 为什么禁止 fallback 标识

1. fallback 会导致同一用户在不同环境/时间产生不同标识，破坏用户身份一致性
2. fallback 数据可能被写入生产数据库，造成脏数据
3. 掩盖了凭证配置问题，增加排查难度

## 二、输入过滤与 XSS 防护

用户可输入的自由文本（昵称、备注、描述等）：

- 过滤控制字符（U+0000-U+001F，不含 U+000A/LF）
- 过滤 HTML 标签（`<[^>]*>`），防止存储型 XSS
- 过滤后与原文不一致则拒绝，返回「输入包含无效字符」
- 输出端再做一次转义（纵深防御，不依赖单一过滤层）

## 三、Session / Token 管理

- Session TTL：按业务安全等级设定（默认 30 天，高敏感场景缩短）
- 定时清理过期与已吊销的 session（已过期超过 N 天 + 已吊销超过 N 天）
- 吊销机制：用户登出 / 改密 / 异常登录 → 立即吊销所有 session

## 四、luban 双后端安全一致性

Java 和 Go 后端的认证逻辑必须**行为一致**（见 `docs/DUAL_BACKEND_PARITY.md`）：

- 同一 token 在 Java 和 Go 解析出的用户身份必须一致
- 同一权限点（perm）在两端的拒绝/放行结果必须一致
- 错误码（401/403）与响应体格式必须一致

## 五、多端安全一致

electron/flutter/web 客户端：

- token 存储方式按端特性（web 用 httpOnly cookie 或内存；electron/flutter 用安全存储）
- token 刷新逻辑三端一致
- 登出时三端都清理本地凭证

## 六、引擎与物料的权限边界

- 引擎渲染的页面，其数据访问权限由 BFF/后端强制校验，**不信任**前端传入的权限标识
- 物料的 props schema 不得包含可被注入的危险字段（如 `innerHTML` 未转义）
- 引擎产物在 SSR（website）及各端渲染时，权限校验逻辑必须一致
