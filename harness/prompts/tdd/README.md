# tdd/

测试驱动开发（TDD）提示词模板。用于：严格走 **红 → 绿 → 重构** 循环来交付一个功能点。

## 循环

```
红：写一个会失败的测试（描述期望行为）
  ↓
绿：写**最少**的代码让测试通过（允许丑陋）
  ↓
重构：在测试持续绿的前提下，清理代码
  ↓
（回到红，加下一个测试）
```

## 何时用

- 改动有明确的「输入→输出」契约（如：引擎 schema 校验函数、BFF 聚合接口、后端某个 service 方法）。
- 想把覆盖率 / 分支覆盖做扎实，而不是事后补测试。
- 修复一个 bug：先写一个**能稳定复现** bug 的失败测试，再修。

## 何时**不**用

- 纯探索性、连契约都还没定 → 先 `prototype/` 或 `brainstorming/`。
- UI 视觉调整（颜色、间距）→ TDD 收益低，用视觉对照 / 截图 diff。

## 反模式（禁止，与 luban E2E 契约一致）

- ❌ 写完实现再补测试，冒充 TDD。
- ❌ 测试用 mock 数据「假装通过」，真实链路没跑。
- ❌ `test.skip` / 条件跳过来换取「绿」（违反 luban E2E 执行契约）。
- ❌ 绿阶段写过度的代码（违反「最少代码」原则）。

## 双后端的特殊用法

luban 有 Java / Go 双后端，TDD 可用作**契约对齐工具**：

1. 先写一份接口契约（请求/响应/错误码）。
2. Java 端 TDD：红（契约测试）→ 绿（实现）。
3. Go 端 TDD：用**同一份契约测试**（语言无关的请求/断言）→ 绿。
4. 两端测试同绿 = 行为一致（满足 CLAUDE.md 硬约束 #3）。
