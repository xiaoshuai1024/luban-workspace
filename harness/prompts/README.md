# harness/prompts

luban-workspace 的「会话级提示词模板」库。每个子目录对应一种工作模式，存放可被 agent / 用户直接复用的 prompt 骨架与产出契约。

> 这些模板是与 `.agents/commands/`（slash 命令）、`.agents/skills/`（技能）互补的**纯文本 prompt 资产**——不依赖任何运行时加载机制，复制粘贴或 source 即可使用。

## 目录

| 子目录 | 用途 | 何时用 |
|--------|------|--------|
| `prototype/` | 快速原型 / Spike | 验证一个想法是否可行（渲染一致性、双后端契约、引擎物料接入） |
| `plan/` | 实现方案 / Plan | 大规模编码前，把范围、阶段、依赖、验收写成可对齐的文档 |
| `brainstorming/` | 头脑风暴 / 讨论稿 | 第一轮发散：列假设、待确认问题、明确不做 |
| `tdd/` | 测试驱动开发 | 红→绿→重构循环，先写失败测试再补实现 |

## 与 kangdou 的差异

- 不绑定 GStack / 云效 / 微信合规语境；聚焦 luban 低代码引擎 + 双后端（Java/Go）。
- plan 模板弱化了 kangdou 的「商详/营销/履约」业务章节，强化「双后端行为一致」「物料 schema 合规」「SSR/多端渲染一致」。
- 每个子目录至少有一个 `README.md`（说明 + 触发场景）和 / 或主模板 `.md`（可直接复用的 prompt 骨架）。

## 使用约定

1. **先讨论稿，后定稿**：plan / brainstorming 类工作，第一轮只对齐范围与假设，用户拍板后再产出可落盘文档。
2. **复制即用**：模板中的 `<占位符>` 用实际包路径 / 分支名 / 需求标题替换。
3. **不写业务硬编码**：模板保持通用，业务细节由调用方填入。
