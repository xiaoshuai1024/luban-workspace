## 第二轮：定稿（用户拍板后再产出）

把定稿写入 `docs/superpowers/plans/<feature>.md`，须包含：

- 文首 YAML：`taskGraph: docs/superpowers/tasks/<featureId>.json`（同一批次创建/更新该 JSON）
- §0 分支与阶段：方案编写阶段分支 + 执行阶段 `feature/<主题>-<标识>`（多系统同名分支）
- §1 范围与目标（来自讨论稿，用户确认后的版本）
- §2 双后端契约（Java/Go 响应体/错误码/状态机一致性要求）
- §3 业务逻辑（引擎/物料/BFF/后端各自要落的关键逻辑）
- §4 实现步骤
  - §4.0 新增模块/文件清单
  - §4.1 UX 自检摘要（若涉及用户可见流程）
  - §4.2 列表级交互链（关键路径 step-by-step）
  - §4.3 页面结构 / 物料接入
- §5 测试计划
  - 单测（覆盖率目标：TS 引擎/bff/website 85% · UI 90% · Java 80% · Go 75%）
  - E2E（禁止 skip，真实执行）
  - 双后端对齐验证
- §6 验收门禁
  - `make test-coverage` 通过
  - 引擎产物在 web(SSR) 及各端渲染一致
  - 物料 props schema 合规
- §7 风险与回滚

## 硬约束（MUST）

- 信息与代码必须真实，禁止推测 API/签名。
- 改码前必须 Read 确认；3+ 文件或跨子仓改动先列范围等用户确认。
- 文件 UTF-8 without BOM。
- 合并冲突优先分析双方逻辑保留双方，禁止 --ours/--theirs。
